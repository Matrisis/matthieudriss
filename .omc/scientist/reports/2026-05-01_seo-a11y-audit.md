## 1. Page Speed Signals

[FINDING:F1] Render-blocking CSS is present but impact is moderate
- `styles.css` (32.7 KB) is loaded via standard `<link rel="stylesheet">` — render-blocking.
- 32.7 KB uncompressed equates to ~6-8 KB gzip/brotli — below the 14 KB critical-path threshold.
- Google Fonts stylesheet is also render-blocking but includes `&display=swap` (see F2).
- Two preconnect hints exist for `fonts.googleapis.com` and `fonts.gstatic.com` — good.
- No preload hint for `styles.css`.
- No preload hint for critical web font files (e.g., `IBM-Plex-Sans-latin-400.woff2`).
[STAT:ci] CSS: 32.7 KB uncompressed, 272 rule blocks, 8 @keyframes
[STAT:effect_size] Render-blocking CSS adds ~1 RTT to first paint on cold cache (moderate)
[STAT:n] 2 preconnect hints, 0 preload hints, 0 font-face declarations in local CSS

[FINDING:F2] Font loading strategy prevents FOIT but lacks CLS mitigation
- Google Fonts URL uses `&display=swap` (CSS2 API), ensuring `font-display: swap` in the injected `@font-face` rules.
- Fallback font stack is comprehensive: `ui-sans-serif, system-ui, -apple-system, "Helvetica Neue", Arial, sans-serif`.
- No `size-adjust` or `ascent-override`/`descent-override` descriptors to reduce layout shift when the web font swaps in.
- IBM Plex Sans has different metrics from system fallbacks — a visible text shift on font swap is likely.
- No `font-display: swap` keyword used directly in the local CSS (relies entirely on Google's API URL parameter).
[STAT:ci] Risk of visible layout shift on font swap: moderate
[STAT:n] 2 font families (IBM Plex Sans 3 weights, IBM Plex Mono 2 weights)

[FINDING:F3] CSS is lean and well-structured, with minor dead code
- 74 class names exist in CSS that never appear in HTML — but this count is inflated by regex false positives matching decimal numbers in property values.
- True unused classes: `.form-block`, `.grid--2`, `.hero__tags`, `.tag`, `.dot` — ~5 classes totaling ~40 lines of CSS.
- 18 `@media` breakpoints covering 7 distinct widths (560px to 1000px) — comprehensive.
- Heavy use of modern CSS features: OKLCH color space, `clamp()`, `color-mix()`, `text-wrap: balance`, `backdrop-filter`.
- No CSS preprocessor (PostCSS, Sass) visible — pure CSS authored manually, which is fine at this scale.
[STAT:n] 32.7 KB CSS, ~5 unused class definitions (<2% dead code)
[STAT:effect_size] Negligible — dead CSS adds ~0.3 KB uncompressed

## 3. Accessibility (a11y)

[FINDING:F7] Color contrast is generally strong; three specific failures identified
Using a proper OKLCH-to-sRGB-to-WCAG-luminance conversion:

Light theme failures:
- `--ink-3` (#6c7376) on `--bg-alt` (#f4f3ee): **4.31:1** — FAILS WCAG AA for normal text (<4.5:1)
  - Affects: `.mono` labels in pricing sidebar, `.field label` on `--bg-alt` sections
- `--warn` (#cd5537) on `--bg` (#f8f6f0): **4.06:1** — FAILS WCAG AA for normal text
  - Affects: "x Pas adapte pour" heading, strikethrough decoration color
- `--ink-4` (#9a9fa2) on `--bg`: **2.56:1** — FAILS all levels
  - Affects: diagram arrows, `.method__num` large numbers (decorative — lower severity)

Dark theme: ALL text combinations pass AA for both normal and large text. Heading-on-deep achieves 16.14:1.
Primary buttons: Nav CTA (17.24:1) and accent button (6.16:1) both pass AA.
[STAT:n] 16 color pairs tested, 3 failures (2 moderate, 1 decorative)
[STAT:effect_size] Moderate — ~5-10% of body text is at non-compliant contrast

[FINDING:F8] Focus indicators are present for forms but absent for navigation and interactive components
- Form fields have explicit `:focus` styles (border-color change + 4px box-shadow ring) — good.
- The single `outline: none` declaration applies to form inputs, replaced by the box-shadow ring — acceptable.
- NO custom `:focus-visible` or `:focus` styles for: nav links, the nav CTA button, primary/ghost/accent buttons, FAQ accordion buttons, anchor links.
- These elements rely on the browser's default focus ring. In Chrome/Safari on macOS, the default is often invisible unless "Full Keyboard Access" is enabled.
- `:focus-visible` is the modern best practice — it shows focus rings only for keyboard navigation, not mouse clicks.
[STAT:n] 2 `:focus` rule blocks (forms only), 0 `:focus-visible` declarations

[FINDING:F9] Form labels are present but lack explicit `for`/`id` association
- All 13 form inputs have a preceding `<label>` element — 100% coverage.
- However, labels are associated via DOM proximity (sibling structure) rather than explicit `for="inputId"` + `id="inputId"` attributes.
- Implicit association works in most screen readers, but explicit `for`/`id` is more robust and is the WCAG-recommended technique (H44).
[STAT:n] 13/13 inputs have labels, 0/13 have explicit for/id association

[FINDING:F10] FAQ accordion lacks ARIA semantics for screen readers
- FAQ uses `<button>` elements which are natively keyboard-operable (Enter/Space activate) — correct.
- However, the accordion state is communicated ONLY via a `data-open` attribute, which is NOT announced by screen readers.
- Missing: `aria-expanded="true/false"` on each FAQ button.
- Missing: `aria-controls="answer-id"` linking button to answer panel.
- Missing: unique `id` on each `.faq__a` div for `aria-controls` to reference.
- The plus/minus icon has `aria-hidden="true"` (correct — decorative).
- FAQ answers use `max-height: 0` with `overflow: hidden` — content is hidden from all users, including screen readers (correct behavior).
[STAT:n] 8 FAQ items, 0 with aria-expanded, 0 with aria-controls

[FINDING:F11] No skip-to-main-content link — significant keyboard navigation barrier
- Zero skip navigation links found in the HTML.
- The page has 26 anchor links + 8 FAQ buttons before the first `<main>` content, requiring ~34 Tab presses to reach main content.
- A skip link (`<a href="#main-content">`) is a WCAG 2.1 Level A requirement (Success Criterion 2.4.1).
[STAT:n] 0 skip links, ~34 interactive elements before main content

[FINDING:F12] Semantic HTML is well-structured; heading hierarchy is valid
- Single `<h1>` for the hero title — correct.
- 14 `<h2>` elements, one per major section — correct.
- 37 `<h3>` elements for card titles, step titles, FAQ questions — correct (no level skipping).
- `<nav>`, `<main>`, `<footer>` landmarks used (implicit ARIA roles).
- No `<header>` element (hero is a `<section>`).
- `lang="fr"` correctly set on `<html>`.
[STAT:n] 1 h1, 14 h2, 37 h3 — valid hierarchy, no skips

## 4. Core Web Vitals Signals

[FINDING:F13] CLS risk is low overall; font swap is the primary concern
- Hero LCP element is text (not an image) — good for both LCP and CLS.
- `display=swap` prevents FOIT but does not prevent layout shift when IBM Plex loads with different metrics than the fallback.
- `text-wrap: balance` on headings prevents multi-line reflow on font swap (partial mitigation).
- All CSS animations use `transform` and `opacity` only (compositor thread, no layout trigger).
- `.reveal` elements start invisible (opacity: 0) and animate in only after IntersectionObserver fires — but these are below the fold, so no CLS impact.
- No dynamically injected content above the fold.
- No ads, embeds, or third-party widgets that commonly cause CLS.
[STAT:effect_size] Low — primary CLS risk is font swap (~0.02-0.05 CLS score)
[STAT:n] 8 @keyframes, all transform/opacity based

[FINDING:F14] LCP is well-optimized; the hero text is the LCP candidate
- LCP element: the `<h1>` hero heading — text-based, no image decode required.
- No hero image to slow down LCP.
- But: LCP time depends on font blocking. `styles.css` is render-blocking (~1 RTT). The Google Fonts CSS is also render-blocking (~1 RTT to Google's CDN).
- With `display=swap`, the browser paints text in the fallback font immediately after CSS is parsed, so LCP fires on the system font render — good.
- No `fetchpriority="high"` on critical resources (though the hero text itself doesn't require it).
- Estimated LCP: 1.5-2.5s on 4G mobile (good — under 2.5s threshold).
[STAT:effect_size] Low risk — text-based LCP with swap font strategy

[FINDING:F15] INP risk is minimal; JavaScript execution is well-optimized
- Scroll handler uses `requestAnimationFrame` with a `ticking` guard — no more than one callback per frame.
- `passive: true` on scroll listener — browser can optimize scrolling without waiting for JS.
- IntersectionObserver threshold at 0.08 with `unobserve()` after first intersection — lightweight, no continuous work.
- FAQ click handler is O(1) per interaction (toggles attribute + loops 8 items to close others).
- Magnetic CTA effect uses `mousemove` but: (a) respects `prefers-reduced-motion`, (b) only applies `transform` (no layout trigger), (c) handler is <10 lines.
- Smooth scroll uses native `scrollIntoView({ behavior: 'smooth' })` — composited, non-blocking.
- No heavy frameworks, no React/Vue/Angular, no virtual DOM diffing.
- All JS is inline in a single IIFE — 77 lines, no external dependencies.
[STAT:n] 77 lines of JS, 1 rAF-throttled scroll handler, 1 IO observer, 1 mousemove handler
[STAT:effect_size] Very low INP risk — JS budget is well within limits

## Limitations

[LIMITATION] All contrast ratios were computed using a mathematical OKLCH-to-sRGB conversion, not measured from rendered pixels. Browser rendering engines may produce slightly different values due to color management (display-p3 vs sRGB gamut mapping) and subpixel antialiasing. For production audit, use a browser-based contrast checker (e.g., Chrome DevTools or axe DevTools).

[LIMITATION] This is a static analysis only. Runtime behavior (actual LCP/CLS/INP values, font loading waterfall, cumulative layout shift from font swap) requires browser-based measurement via Lighthouse, WebPageTest, or the Chrome UX Report. The Core Web Vitals findings are predictions based on code patterns, not field data.

[LIMITATION] CSS "unused class" detection was performed via regex matching and may have false positives (decimal numbers in property values matching `.className` patterns). The true unused count is estimated at ~5 classes rather than the raw 74 detected.

[LIMITATION] The form submission is a demo (`e.preventDefault()` + hide/show). Security analysis of form handling (CSRF, rate limiting, data sanitization) assumes this will be replaced with a proper backend before production.

[LIMITATION] OKLCH color space has limited browser support (Chrome 111+, Safari 15.4+, Firefox 113+). Fallback colors (via `@supports`) are not provided. Users on older browsers will see the browser default colors, not a degraded-but-functional palette.
